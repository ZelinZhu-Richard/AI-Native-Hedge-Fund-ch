# Day 27 Plan

## Goal

Add a grounded reporting layer that helps operators, researchers, and reviewers inspect the system without replacing source truth.

## What Was Added

- new reporting schemas in `libraries/schemas/reporting.py`
- new `ReportingService` in `services/reporting/`
- dedicated `artifacts/reporting/` root in the shared artifact workspace
- persisted report artifacts:
  - `research_summaries/`
  - `risk_summaries/`
  - `review_queue_summaries/`
  - `experiment_scorecards/`
  - `proposal_scorecards/`
  - `daily_system_reports/`
  - `system_capability_summaries/`

## Integrated Paths

- backtest workflow now emits `ExperimentScorecard`
- strategy ablation now emits `ExperimentScorecard`
- portfolio review pipeline now emits `RiskSummary` and `ProposalScorecard`
- operator review context now loads latest available grounded summaries and scorecards for proposal review
- daily orchestration operations summary now emits `ReviewQueueSummary` and `DailySystemReport`

## Grounding Rules

- summaries are deterministic
- no LLM prose generation was introduced
- every report links back to source artifacts
- warnings, missing information, and refusal or quarantine artifacts stay visible
- scorecards expose measured dimensions instead of one opaque score

## Limits

- reporting does not replace validation, evaluation, or review gates
- research summaries are service-supported but not yet auto-generated by the research workflow
- no dashboard or new API surface was added
- latest-artifact selection is still a separate problem from report grounding

## Verification

The Day 27 implementation is expected to keep:

- `ruff check`
- `mypy`
- `pytest -q`

green after the reporting integrations and new tests land.

## Best Follow-On

The highest-leverage next step is to make Week 4 policy consume `ProposalScorecard`, `ExperimentScorecard`, `RiskSummary`, `ValidationGate`, and open paper-ledger followups together, so summary artifacts become policy-useful without becoming the new source of truth.
