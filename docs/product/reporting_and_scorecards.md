# Reporting And Scorecards

## Purpose

Day 27 adds a deterministic reporting layer for operators, researchers, and reviewers.

These reports are not a dashboard replacement and they are not source truth. They are persisted summary artifacts built from existing typed artifacts so people can inspect the system without reading every raw object by hand.

Reporting artifacts live under `artifacts/reporting/`.

## What Exists

The reporting layer currently produces:

- `ResearchSummary`
  - grounded summary of one `ResearchBrief` plus linked hypothesis, counter-hypothesis, evidence assessment, warning artifacts, and refusal or quarantine artifacts
- `RiskSummary`
  - grounded summary of one `PortfolioProposal` plus risk checks, stress results, validation gates, and reconciliation context
- `ReviewQueueSummary`
  - queue snapshot with counts, escalations, unassigned items, and attention-bearing items
- `ExperimentScorecard`
  - explicit scorecard for one experiment and its evaluation or realism context
- `ProposalScorecard`
  - explicit scorecard for one portfolio proposal and its construction, risk, stress, reconciliation, and paper-trade context
- `DailySystemReport`
  - operator-facing daily report combining monitoring, review queue, paper-ledger summaries, open followups, and recent scorecards
- `SystemCapabilitySummary`
  - grounded capability snapshot for one subsystem or workflow family

## Audiences

- Operators
  - `ReviewQueueSummary`
  - `DailySystemReport`
- Researchers
  - `ResearchSummary`
  - `ExperimentScorecard`
- Reviewers
  - `RiskSummary`
  - `ProposalScorecard`
- System and architecture review
  - `SystemCapabilitySummary`

## Grounding Rules

Every reporting artifact is built from persisted structured artifacts only.

Grounding is preserved through:

- explicit `source_artifact_ids`
- explicit `warning_artifact_ids`
- explicit missing-information lists
- linked refusal or quarantine `ValidationGate` IDs where relevant
- report provenance that names the reporting transformation and upstream artifact IDs

`ReportingContext` is a derived trace bundle that explains what a report was built from. It does not replace the underlying artifacts.

## Scorecard Design

Scorecards do not emit a single readiness score.

Instead they expose explicit `ScorecardMeasure` rows with:

- `measure_name`
- `measure_basis`
- explicit `pass | warn | fail` status
- linked artifact IDs
- notes

Current experiment scorecard measures include:

- experiment metadata integrity
- evaluation coverage
- failure-case pressure
- robustness pressure
- execution realism gap

Current proposal scorecard measures include:

- construction traceability
- risk-check pressure
- stress fragility
- reconciliation alignment
- paper-trade materialization

## Workflow Integration

The reporting layer is currently attached to these real paths:

- backtest workflow
  - emits and persists `ExperimentScorecard`
  - appends the scorecard back into experiment context
- strategy ablation workflow
  - emits and persists `ExperimentScorecard`
  - appends the scorecard back into parent experiment context
- portfolio review pipeline
  - emits and persists `RiskSummary`
  - emits and persists `ProposalScorecard`
  - stamps `proposal_scorecard_id` onto the persisted proposal
- operator review context
  - loads latest available `ResearchSummary`, `RiskSummary`, and `ProposalScorecard`
- daily orchestration operations summary
  - emits and persists `ReviewQueueSummary`
  - emits and persists `DailySystemReport`

## Limitations

- reports are deterministic summaries, not analytical inference engines
- no free-form LLM summary generation is used here
- `ResearchSummary` is available through the service, but it is not yet auto-generated by the research workflow
- `DailySystemReport` depends on what upstream workflows persisted; it does not invent coverage
- scorecards do not replace policy gates, validation gates, or evaluation artifacts

## What This Does Not Claim

- no composite readiness score
- no alpha score
- no automatic approval recommendation
- no guarantee that a green scorecard means a proposal or experiment is safe to promote

The reporting layer improves digestibility. It does not weaken the requirement to inspect source artifacts when risk, evaluation, or validation questions matter.
